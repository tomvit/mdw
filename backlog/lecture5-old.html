Service Usage Process

Goal and Service Use Tasks

  • Goal (represents a requester entity)
    • a requester's desired state (described by an effect)
      • functional: to buy and deliver a book
      • non-functional (preferences): price; quality
      • behavioral: (public process) provides instance data
        e.g., a Harry Potter book; Innsbruck, AT
  • Service Use Tasks
    • investigate whether and how to use services to fulfill the goal
    • discovery: a service that matches an abstract goal
    • negotiation: the service hat can fulfill a goal instance
    • selection: the service that satisfies user's preferences
    • composition: composes services into a process
    • mediation: data and process interoperability
    • invocation: invokes the service

Usage Process

Usage Process Implementations

  • Manual
    • Handful number of services
    • binding usually manual (~ design time)
      • tool support for individual tasks (such as discovery)
    • Execution
  • Automated
    • intelligent services – large number of services
    • binding = late-binding (~ runtime)
    • need for formal descriptions with high expressivity
      • logic-based descriptions, Semantic Web
      • balance between manual and automated tasks
    • user should always be in the loop to approve, relax requirements, etc.

Service Description and Use Tasks

Types of Services

  • End-user services
    • by service we mean end-user service by default
    • they bring some business value to their users
    • also called business services
    • composed into business processes
  • Middleware Services
    • they facilitate end-user service usage process
    • also called platform services
    • composed into middleware processes
      → late-binding, execution

Discovery

Discovery Types

  • Discovery by types of service descriptions
    • Non-functional descriptions
      • To discover services by name, author, description, price, etc.
    • Functional descriptions – capability
      • To discover services that match requested capability, that is preconditions and effects from the Goal
      • This requires a logical system (to evaluate logical conditions)
    • Functional descriptions – functional category
      • To discover services that match a requested functional category
  • Terminology in the research community
    • Not a common agreement on terms
    • Sometimes discovery ~ abstract discovery + negotiation + selection
      • Abstract discovery only operates on abstract descriptions
      • negotiation ~ instance discovery (negotiation implies more sophistication)

Functional Category Discovery

  • Taxonomy
    • A tree where nodes are categories and edges sub-class-of relations.
  • Discovery on functional category
    • To test whether a service category belongs to a sub-tree
    • The sub-tree is defined by a goal category
  • Example taxonomy in xml
    • cf. functional category in Lecture 4

Service Taxonomy and Discovery Illustration

  • Discovery task

Relaxing and Restricting Goal Requirements

  • Relaxation vs. restriction
    • Relaxation – discovery finds more services with general functionality.
    • Restriction – discovery finds less services with specific functionality.

Relaxed/Restricted Discovery

  • Task
    • Change the algorithm so that based on the first discovery result, it returns the new result when performing relaxation or restriction.
  • Description of a solution
    • Let:
      • D be a set of services in a directory
      • t be a service taxonomy root
      • \math{goal} be a goal category
      • R be a set of discovered services such that R = \mathit{discover}(\mathit{goal}, D, t)
      • \math{goal_1} be a goal category that is a relaxation or a restriction of \mathit{goal}
    • Relaxation
      • \mathit{goal}_1 \supset \mathit{goal}\Rightarrow R_1 = \mathit{discover}(\mathit{goal}_1, D-R, t) \cup R
    • Restriction
      • \mathit{goal}_1 \subset \mathit{goal}\Rightarrow R_2 = \mathit{discover}(\mathit{goal}_1, R, t)
Mediation

Interoperability and Mediation

  • Interoperability = ability of two or more services to communicate
    • Data interoperability
      • data syntax (encoding, representations), semantics
    • Operations/functions and process interoperability
      • operations and public processes
    • Technical interoperability
      • application protocols, technical aspects
  • Solutions for interoperability
    • Common interface
      • services agree on an interface they will use
    • Mediation
      • data mediator – transformation of data
      • process mediator – adjustment of services' public process

Solutions for Interoperability

Examples Standard Common Interface

  • Data
    • Encodings such as UTF-8
    • Representations such as XML and JSON
    • Structure (semantics) such as XML Schema
    • Challenges:
      • Domain specific semantics
      • Attempts to standardize such as RosettaNet
      • single vendor solution
  • Operations
    • message-oriented view (~SOAP): domain-specific names
      • For example getOrder, openOrder
      • May add additional complexity to mediation
    • resource-oriented view (~REST): standard CRUD names
      • For example GET {order-id}

Examples Standard Common Interface (Cont.)

  • Processes
    • domain-specific
    • Attempt to standardize by RosettaNet (Partner Interface Processes – PIP), single vendor solution
  • Technical
    • TCP/IP is the common technical interface
    • Standard application protocol is today HTTP
      • At the bottom of many protocols
    • Challenges
      • Many application protocols such as XML-RPC, RMI
      • SOAP is an attempt to build a common technical interface for communication at the services layer

Data Mediation (structure/semantics)

  • There are two phases
  • Definition of mappings (design-time)
    • A mapping associates one data structure to another data structure and defines a conversion between them.
    • Mapping languages
      • declarative/graphical
      • procedural – XSLT, XQuery or 3rd gen. languages (Java, JavaScript, ...)
  • Execution of mappings (runtime)
    • application of mappings to instance data

Entities and Properties

  • Entities and properties
    • Data entity is a set of properties
    • Example: employee is an entity with properties name, salary
      • employee(name, salary)
      • employee.name, employee.salary
  • Implicit and explicit context
    • example explicit context:
      • currency part of a salary property such as salary_usd or
      • part of a salary entity such as salary(currency, value)
    • example implicit context: currency part of a salary property salary.
      • A human must provide the implicit context during the definition of mappings.
  • Types of mappings
    1. property-property
    2. entity-entity
    3. entity-property and property-entity

Property–property mapping

  • Let
    • A, B, C be three properties and a \in A, b \in B, c \in C be their values.
  • Property–property mappings
    • 1. \mathit{pp}_{A \to B}(a) = b is a simple mapping.
    • 2. \mathit{pp}_{AB \to C}(\{a,b\}) = c, \mathit{pp}_{A \to BC}({a}) = \{b,c\} are composed mappings.
    • 3. \mathit{pp}_{A \to B}(a,k) = b is a form of a simple mapping called conversion where k is a conversion value.
    • Example:
      •                                 simple:     nickname --> alias, rank --> rank
                                composed:   address --> { street, city }, { street, city } --> address
                                conversion: currency_pounds --> currency_usd, salary --> salary_eur
  • Idempotent mapping
    • A mapping applied on the same values has always the same result.
    • → Simple mappings should always be idempotent
  • Non-idempotent mapping
    • A result of a mapping applied on the same input changes over time.
    • → currency conversion can be non-idempotent

Property–property mapping (Cont.)

  • Decomposition of mappings
    • \mathit{pp}_{A \to BC}(a) = \mathit{pp}_{A \to B}(a) \cup \mathit{pp}_{A \to C}(a)
    • Example
      •                                 address --> { street, city } => address --> street, address --> city
    • But \mathit{pp}_{AB \to C}(a,b) \neq \mathit{pp}_{A \to C}(a) \cup \mathit{pp}_{B \to C}(b) (would lose a context).
    • Example
      •                                 { street, city } --> address => { street } --> address, { city } --> address
  • Non-loss mapping
    • For two mappings \mathit{pp}_{A \to B} and \mathit{pp}_{B \to A} it holds that \mathit{pp}_{B \to A}(\mathit{pp}_{A \to B}(a)) = a
    • → Simple mappings should always be non-loss
  • Loss mapping
    • For two mappings \mathit{pp}_{A \to B} and \mathit{pp}_{B \to A} it holds that \mathit{pp}_{B \to A}(\mathit{pp}_{A \to B}(a)) \neq a
    • → Conversion mapping can be loss, such as currency conversion

Entity–entity mapping

  • Let
    • X, Y, Z be three entities and x \in X, y \in Y, z \in Z be their properties.
  • Entity–entity mappings
    • 4. \mathit{ee}_{X \to Y} : \forall y \in Y \; \exists x \in X \; \mathit{pp}_{x \to y} is a simple mapping
    • 5. \mathit{ee}_{X \to \mathit{YZ}} : \forall y \in Y \land \forall z \in Z \; \exists x \in X \; \mathit{pp}_{x \to \mathit{yz}}
          \mathit{ee}_{\mathit{XY} \to Z} : \forall z \in Z \; \exists w \in \{X \cup Y\} \; \mathit{pp}_{w \to \mathit{z}}
          are composed mappings
    • Examples:
      •                                 simple:     
                                    employee(name, id, address) --> staff(fname, staffid)
                                        employee.name    --> staff.fname
                                        employee.id      --> staff.staffid
                                
                                composed:
                                    employee(name, id, address) --> { staff(fname, staffid), addr(staffid, str) }
                                        employee.name    --> staff.fname
                                        employee.id      --> staff.staffid
                                        employee.id      --> addr.staffid                                        
                                        employee.address --> addr.str

Entity–entity mapping (Cont.)

  • Idempotent and non-idempotent mapping
    • An entity-entity mapping is idempotent if all its property-property mappings are idempotent . Otherwise it is a non-idempotent mapping.
  • Decomposition of mappings
    • Analogical to decomposition of property-property mappings
  • Non-loss and loss mappings
    • An entity-entity mapping is non-loss if all its property-property mappings are non-loss. Otherwise it is a loss mapping.

Property–entity/entity–property mapping

  • Let
    • X, Y be two entities and A, B two properties such that A \notin X,A \notin Y,B \notin X,B \notin Y.
  • Property–entity/entity–property mappings
    • 6. \mathit{pe}_{A \to X} : \forall x \in X \; \mathit{pp}_{A \to x} is a simple property–entity mapping
    • 7. \mathit{ep}_{X \to A} : \exists x \in X \; \mathit{pp}_{x \to A} is a simple entity–property mapping
    • 8. \mathit{pe}_{A \to \mathit{XY}} : \forall w \in \{X \cup Y\} \; \mathit{pp}_{A \to w} is a comp. property–entity mapping
    • 9. \mathit{ep}_{X \to \math{AB}} : \exists x \in X \; \mathit{pp}_{x \to A} \land \mathit{pp}_{x \to B} is a comp. entity–property mapping
    • Examples:
      •                                 simple property-entity and entity-property:     
                                    address --> addr(street, city)
                                        address    --> addr.street
                                        address    --> addr.city

                                    address(street, city) --> city
                                        address.street    --> city
  • Idempotence and loss/non-loss mappings
    • They are analogical with entity-entity/property-property mappings
